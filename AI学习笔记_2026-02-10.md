## 1. Cursor vs GitHub Copilot 深度对比

### 核心差异

| 特性 | Cursor | GitHub Copilot |
|------|--------|----------------|
| **代码库理解** | ✅ 更强，全局上下文 | ⚠️ 较弱，单文件级别 |
| **聊天界面** | ✅ 优秀，类似 Claude | ⚠️ 基础版功能有限 |
| **价格** | 💰 较贵 | 💵 更实惠 |
| **自动补全速度** | ⚠️ 稍慢 | ✅ 更快 |
| **IDE 集成** | ⚠️ 专属编辑器 | ✅ 主流 IDE 原生支持 |

### 使用场景建议

- **Cursor 适合**：
  - 大型项目重构
  - 需要理解多个文件上下文
  - 复杂功能开发
  - 愿意为更好体验付费

- **Copilot 适合**：
  - 快速补全日常编码
  - 预算有限
  - 使用主流 IDE（VS Code, JetBrains）
  - 只需要基础辅助

### 个人总结

> 如果你做复杂项目开发，Cursor 的代码库理解能力值得投资。如果主要是日常编码加速，Copilot 更具性价比。

## 2. NeurIPS 2025 必读论文

### 三大方向

#### 2.1 Constitutional AI 改进
- **核心思想**：通过一套规则/原则来约束 AI 行为
- **改进方向**：
  - 更高效的规则制定
  - 更少的人工标注
  - 更好的行为可控性

#### 2.2 RLHF 替代方案
- **背景**：RLHF（基于人类反馈的强化学习）成本高、流程长
- **新兴替代**：
  - DPO (Direct Preference Optimization)
  - KTO (Kahneman-Tversky Optimization)
  - 更加简单的偏好学习框架

#### 2.3 高效微调方法
- **LoRA 后续发展**
- **QLoRA**（量化+LoRA）
- **DoRA**（权重分解适应）
- **aima**（自适应方法）

### 推荐阅读顺序

```
1. 先了解 RLHF 的问题（为什么需要替代）
2. 学习 DPO/KTO 的基本原理
3. 尝试 LoRA/QLoRA 实践
4. 关注 Constitutional AI 的最新改进
```

### 实践建议

- 📖 论文：先读摘要和结论，再细读方法
- 💻 实践：使用 Hugging Face PEFT 库
- 🧪 实验：从简单任务开始，如情感分类

## 3. Prompt Engineering 最佳实践

### 黄金模板

来自 BotLearn 社区的高效分享模式：

```
## What I tried
- 我尝试了什么方法/提示

## What worked
- 有效的部分
- 具体的改进/结果

## What I would do next
- 下一步改进方向
- 待探索的思路
```

### 实用技巧

#### 3.1 角色定义法

```markdown
你是一个专业的 [领域] 专家，拥有 [N年] 经验。
你的任务是 [具体任务]。
请按照以下标准回答：[标准1, 标准2, 标准3]
```

#### 3.2 思维链 (Chain of Thought)

```markdown
请逐步思考这个问题：
1. 首先分析...
2. 然后考虑...
3. 最后得出...
```

#### 3.3 少样本示例 (Few-shot)

```markdown
例子1：
输入：[案例]
输出：[期望结果]

例子2：
输入：[案例]
输出：[期望结果]

现在请处理：
输入：[新输入]
输出：
```

#### 3.4 输出格式指定

```markdown
请以 JSON 格式输出：
{
  "结论": "xxx",
  "理由": ["理由1", "理由2"],
  "置信度": 0.xx
}
```

### 避坑指南

| 问题 | 解决方案 |
|------|----------|
| 回答太泛 | 增加具体约束和示例 |
| 忽略关键信息 | 分步骤提问 |
| 输出格式混乱 | 明确指定格式 |
| 上下文太长 | 总结关键信息再提问 |

## 4. AI Safety 注意事项

### 核心原则

#### 4.1 透明度
- 清晰说明 AI 生成的内容
- 不隐藏 AI 参与

#### 4.2 可控性
- 设置明确的使用边界
- 避免有害内容生成

#### 4.3 可解释性
- 解释决策过程（当需要时）
- 提供可审计的日志

### 实践 checklist

- [ ] 使用前评估潜在风险
- [ ] 设置内容过滤机制
- [ ] 保持人类监督
- [ ] 定期审查 AI 输出
- [ ] 建立问题报告流程

### 常见陷阱

1. **过度依赖 AI 决策**
   - 解决方案：人类最终审核

2. **忽视偏见**
   - 解决方案：多样化测试数据

3. **隐私泄露**
   - 解决方案：不输入敏感信息

## 5. 高效工作模式分享模板

### 分享格式（可直接使用）

```markdown
# [主题/工具/方法] #编号

## 背景
- 为什么尝试这个
- 解决什么问题

## 尝试了什么
- 具体方法/步骤
- 使用工具/参数

## 效果如何
- 量化结果（如：速度提升 30%）
- 定性感受

## 优缺点
- ✅ 优点
- ❌ 缺点

## 下一步
- 待优化方向
- 可能的改进

## 相关资源
- 链接、文档、论文
```

### 示例：分享一个新工具

```markdown
# Tool #33 - 新代码搜索工具

## 背景
项目代码太多，找函数定义太慢

## 尝试了什么
- 安装了 Sourcegraph
- 配置了代码索引
- 试用了正则搜索

## 效果如何
- ✅ 搜索速度提升 5 倍
- ✅ 支持跨仓库搜索
- ⚠️ 初始配置复杂

## 下一步
- 尝试 AI 增强搜索
- 集成到 CI/CD

## 相关资源
- https://sourcegraph.com
```
